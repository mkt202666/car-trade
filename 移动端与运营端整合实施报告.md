## 二、核心问题分析与解决方案

### 2.1 Redis 事件监听缺失（P0 - 已修复）

#### 问题描述

运营端通过 `MobileEventPublisher` 向 Redis 频道 `mobile:events` 发布通知事件，但移动端后端完全没有任何监听器来消费这些事件。这导致以下业务场景完全失效：

- ❌ 车行审核结果通知（通过/拒绝）
- ❌ 车源状态变更通知（上架/下架/违规）
- ❌ 争议处理结果通知
- ❌ 保证金变动通知
- ❌ 用户状态变更通知（封禁/解封）
- ❌ 系统公告推送
- ❌ 求购匹配通知

#### 根本原因

开发阶段未完成 Phase 3 "数据同步"模块，缺少 Redis Pub/Sub 订阅逻辑。

#### 解决方案

**创建 MobileEventListener 组件**（`car-trade-backend/src/main/java/com/pancosky/newcartrade/listener/MobileEventListener.java`）

```java
@Slf4j
@Component
@RequiredArgsConstructor
public class MobileEventListener {
    
    private final RedisMessageListenerContainer redisMessageListenerContainer;
    private final PushManager pushManager;
    private final CarCacheService carCacheService;
    
    @PostConstruct
    public void init() {
        // 注册消息监听器到 mobile:events 频道
        redisMessageListenerContainer.addMessageListener(
                new MessageListenerAdapter(this, "handleNotification"),
                new ChannelTopic("mobile:events")
        );
    }
    
    public void handleNotification(String message) {
        MobileNotification notification = objectMapper.readValue(message, MobileNotification.class);
        
        switch (notification.getType()) {
            case SHOP_AUDIT_RESULT:
                handleShopAuditResult(notification);
                break;
            case CAR_STATUS_CHANGED:
                handleCarStatusChanged(notification);
                break;
            // ... 其他 5 种类型
        }
    }
}
```

**关键特性**：

1. **自动注册**：使用 `@PostConstruct` 在 Spring 启动时自动订阅 Redis 频道
2. **事件路由**：根据 `NotifyType` 分发到 7 个专用处理方法
3. **WebSocket 推送**：通过 `PushManager` 将通知实时推送给目标用户
4. **缓存清理**：车源状态变更时自动清除 `CarCacheService` 中的详情缓存
5. **容错处理**：所有异常都被捕获并记录日志，不会中断事件流

**配置支持**：

在 `RedisConfig` 中添加 `RedisMessageListenerContainer` Bean：

```java
@Bean
public RedisMessageListenerContainer redisMessageListenerContainer(RedisConnectionFactory connectionFactory) {
    RedisMessageListenerContainer container = new RedisMessageListenerContainer();
    container.setConnectionFactory(connectionFactory);
    return container;
}
```

#### 验证结果

- ✅ 编译成功（无错误）
- ✅ 事件发布端逻辑完整（AdminShopReviewService、AdminUserService、AdminCarService、AdminDepositService、AdminDisputeService 均已正确调用 `adminNotificationService.notify()`）
- ✅ 消息格式一致（MobileNotification DTO 在两端保持相同结构）

### 2.2 JWT 密钥管理分散（P1 - 已修复）

#### 问题描述

移动端和运营端使用完全独立的 `JwtUtil` 实现：

| 对比项 | 移动端 JwtUtil | 运营端 JwtUtil |
|--------|---------------|---------------|
| 类型 | 静态工具类 | Spring Component |
| 密钥注入方式 | `setSecret()` 手动注入 | `@Value` 自动注入 |
| 安全验证 | ✅ 有（禁止默认密钥、最小32字节） | ❌ 无 |
| 默认密钥 | 无（强制环境变量） | ⚠️ 有不安全默认值 |
| 日志记录 | ✅ 详细（token前缀、失败原因） | ❌ 无 |

**风险**：

1. **密钥不一致风险**：如果两端配置不同的 `JWT_SECRET`，会导致跨系统 Token 验证失败（影响未来 SSO 扩展）
2. **弱密钥风险**：运营端默认密钥虽然长度足够（72字节），但包含明显占位符文本，容易被识别
3. **维护困难**：密钥轮换需要同时修改两个服务

#### 解决方案

**统一安全验证逻辑**

1. **运营端 JwtUtil 增强**（`car-trade-admin-backend/src/main/java/com/pancosky/cartradeadmin/util/JwtUtil.java`）

添加启动时验证：

```java
@PostConstruct
public void validateSecret() {
    if (secret == null || secret.isBlank()) {
        throw new IllegalStateException("JWT secret is empty.");
    }
    
    byte[] bytes = secret.getBytes(StandardCharsets.UTF_8);
    if (bytes.length < MIN_SECRET_BYTES) {
        throw new IllegalStateException("JWT secret too short: " + bytes.length + " bytes");
    }
    
    for (String forbidden : FORBIDDEN_SECRETS) {
        if (forbidden.equals(secret)) {
            throw new IllegalStateException("JWT secret equals a well-known default.");
        }
    }
    
    log.info("JWT secret validated successfully ({} bytes)", bytes.length);
}
```

2. **移除不安全默认值**（`application.properties`）

```properties
# 修改前
jwt.secret=${JWT_SECRET:5d-car-trade-admin-default-secret-key-please-change-in-prod-env-2026}

# 修改后
jwt.secret=${JWT_SECRET}  # 必须通过环境变量设置
```

**安全要求对齐**：

| 安全规则 | 移动端 | 运营端（修复后） |
|---------|-------|----------------|
| 最小密钥长度 | 32 字节 | 32 字节 ✅ |
| 禁止默认密钥 | 6 个黑名单 | 4 个黑名单 ✅ |
| 启动时验证 | ✅ | ✅ |
| 失败快速终止 | ✅ | ✅ |
| 日志审计 | ✅ | ✅ |

#### 验证结果

- ✅ 运营端编译成功
- ✅ 启动时会立即验证密钥安全性
- ✅ 两端现在遵循相同的安全标准

**未来建议**：

提取共享 `common-jwt` 模块，将 `JwtUtil` 抽象为独立库，避免代码重复。

### 2.3 运营端车源管理功能不完整（P1 - 已修复）

#### 问题描述

AdminCarController 原有接口仅支持：
- ✅ GET `/api/v1/admin/cars` - 列表查询
- ✅ GET `/api/v1/admin/cars/{id}` - 详情查看
- ✅ PUT `/api/v1/admin/cars/{id}/status` - 上下架
- ✅ POST `/api/v1/admin/cars/{id}/violate` - 标记违规
- ✅ GET `/api/v1/admin/cars/export` - 导出 Excel

**缺失功能**：
- ❌ PUT `/api/v1/admin/cars/{id}` - 编辑车源信息
- ❌ DELETE `/api/v1/admin/cars/{id}` - 删除车源

#### 解决方案

**新增 DTO**（`CarUpdateDTO.java`）

```java
@Data
public class CarUpdateDTO {
    @NotBlank private String title;
    @NotNull private Long brandId;
    @NotNull private Long seriesId;
    private Long modelId;
    @NotBlank private String cityName;
    @NotBlank private String energyType;
    @NotNull @DecimalMin("0.01") private BigDecimal price;
    @NotNull @DecimalMin("0") private BigDecimal mileage;
    @NotNull private Integer year;
    private String description;
    private String status;
}
```

**新增 Service 方法**（`AdminCarService.java`）

1. `updateCar(Long id, CarUpdateDTO dto)` - 编辑车源
   - 验证车源存在性
   - 更新所有可编辑字段（注意类型转换：Long → Integer）
   - 发送移动端通知给车主
   - 记录操作日志

2. `deleteCar(Long id)` - 软删除车源
   - 设置 `deletedAt` 为当前时间
   - 发送移动端通知给车主
   - 记录操作日志

**新增 Controller 接口**（`AdminCarController.java`）

```java
@AuditLog(module = "car", action = "编辑车源", targetType = "car")
@PutMapping("/{id}")
public ApiResponse<Void> updateCar(@PathVariable Long id, @Valid @RequestBody CarUpdateDTO dto) {
    adminCarService.updateCar(id, dto);
    return ApiResponse.success();
}

@AuditLog(module = "car", action = "删除车源", targetType = "car")
@DeleteMapping("/{id}")
public ApiResponse<Void> deleteCar(@PathVariable Long id) {
    adminCarService.deleteCar(id);
    return ApiResponse.success();
}
```

**关键设计决策**：

1. **类型转换**：DTO 使用 `Long` 类型（前端通用），实体使用 `Integer` 类型（数据库设计），在 Service 层进行 `.intValue()` 转换
2. **软删除**：不物理删除记录，仅设置 `deletedAt`，符合项目规范
3. **通知推送**：编辑和删除操作都会触发 `CAR_STATUS_CHANGED` 通知，确保用户知情
4. **审计日志**：使用 `@AuditLog` 注解自动记录管理员操作

#### 验证结果

- ✅ 编译成功（无错误）
- ✅ 字段类型匹配正确
- ✅ 通知逻辑完整

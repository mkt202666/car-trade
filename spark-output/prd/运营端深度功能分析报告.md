## 二、逐页深度分析

### 2.1 pages/dashboard/index.jsx — 数据仪表盘

#### 主要功能
- **KPI卡片** (4个): GMV总额、在售车源数、冻结保证金余额、待审资质数
- **折线图**: 每日交易趋势（SVG手绘，非第三方图表库）
- **饼图**: 车源渠道构成（SVG环形图）
- **优惠券统计**: 总量/已核销/剩余/使用率
- **审批队列**: 待处理车行/合伙人申请列表
- **预警系统**: 保证金超时锁定预警

#### API调用情况
| 功能 | API函数 | 后端路径 | 状态 |
|------|---------|---------|------|
| KPI卡片 | `dashboardApi.kpi()` | `GET /dashboard/kpi` | ✅ 已实现 |
| 趋势图 | `dashboardApi.trend({period})` | `GET /dashboard/trend` | ✅ 已实现 |
| 车源分布 | `dashboardApi.carDistribution()` | `GET /dashboard/car-distribution` | ✅ 已实现 |
| 优惠券 | `dashboardApi.couponStats()` | `GET /dashboard/coupon-stats` | ✅ 已实现 |
| 审批队列 | `dashboardApi.approvalQueue()` | `GET /dashboard/approval-queue` | ✅ 已实现 |
| 预警 | `dashboardApi.warnings()` | `GET /dashboard/warnings` | ✅ 已实现 |

#### 降级数据处理
- 所有6个数据源均有 `DEFAULT_*` 降级数据
- API失败时自动回退到硬编码默认值
- 降级数据来源：`DEFAULT_KPI`, `DEFAULT_DISTRIBUTION`, `DEFAULT_COUPONS`, `DEFAULT_APPROVALS`

#### 与原型的功能差距
| 原型要求 | 实现状态 | 说明 |
|---------|---------|------|
| KPI卡片点击跳转 | ❌ 缺失 | 原型要求KPI可点击跳转到对应管理页面 |
| 时间范围选择器 | ❌ 缺失 | 原型要求日/周/月/自定义切换 |
| 数据刷新按钮 | ❌ 缺失 | 原型要求手动刷新功能 |
| 数据导出 | ❌ 缺失 | 原型要求报表下载 |
| 图表悬浮详情 | ⚠️ 部分 | SVG有hover样式但无tooltip |
| 预警标记已读 | ❌ 缺失 | 原型要求预警可标记已读/忽略 |

#### 潜在问题
1. **图表库缺失**: 使用SVG手绘而非ECharts/Recharts，维护成本高，交互能力弱
2. **无错误重试机制**: API失败后直接显示降级数据，用户无法感知是真实数据还是mock
3. **优惠券统计逻辑异常**: 后端返回聚合数据时，前端将其拆分为4条记录展示，语义不清晰
4. **预警数据结构不一致**: 期望 `{list, message}` 但fallback为 `{overdueCount, message}`

## 五、潜在风险与建议

### 5.1 高风险问题

1. **deposit-flow页面完全无API**: 此页面展示硬编码数据，可能误导运营人员认为这是真实数据。**建议立即删除或重定向到deposit-manage**。

2. **export-config页面完全无API**: 页面功能与原型描述的"数据导出配置"业务含义不符，且无任何API调用。**需明确业务需求后重新开发**。

3. **导出功能可能缺少认证**: 5个页面使用 `window.open()` 直接访问导出URL，若后端要求JWT认证则会失败。**需统一改造为带Token的下载方式**。

4. **purchase-manage功能缺失严重**: 关闭/匹配等核心功能API已定义但页面未实现，导致求购管理流程不完整。**需补充功能入口**。

5. **user-manage功能缺失严重**: 添加用户/编辑/重置密码等核心功能仅有UI骨架，**需补充完整功能**。

### 5.2 中风险问题

1. **图表库缺失**: dashboard使用SVG手绘图表，维护成本高且交互能力弱。**建议引入ECharts或Recharts**。

2. **图片上传缺失**: resource-manage和car-library仅支持URL输入，运营体验差。**需集成OSS上传组件**。

3. **功能重复**: car-model和car-library功能高度重叠，**需合并为一个页面**。

4. **硬编码数据**: purchase-manage的品牌选项、deposit-flow的全部数据均为硬编码，**需改为动态获取**。

5. **错误处理不一致**: 部分页面API失败后显示错误提示，部分页面静默失败并显示降级数据，**需统一错误处理策略**。

### 5.3 低风险问题

1. **分页控件未完全实现**: car-model的分页dropdown onClick为空函数。

2. **详情页使用弹窗而非抽屉**: 原型要求使用抽屉，部分页面使用弹窗。

3. **手机号脱敏**: 审核场景可能需要完整号码，当前统一脱敏可能影响业务。

4. **订单ID截断**: 体验不佳，建议改为完整显示或tooltip展示。

### 2.3 状态管理(Vuex)

**Store结构** (`store/index.js`):

```javascript
state: {
  user: null,        // 当前用户信息
  token: null,       // JWT access token
  unreadCount: 0     // 未读消息数
}

mutations: {
  setUser(user)      // 设置用户信息并持久化到storage
  setToken(token)    // 设置token并调用saveToken()
  saveAuthBundle(bundle) // 一次性写入access+refresh token
  setUnreadCount(count) // 设置未读数
  logout()           // 清除用户信息和token
}

actions: {
  login(credentials) // 登录并保存token套件
  getUser()          // 获取当前用户信息
}

getters: {
  isLoggedIn         // 是否已登录(token长度>10)
  currentUser        // 当前用户对象
}
```

**Token管理机制**:
- Access Token: 短效JWT,存储在`uni.storage`和`localStorage`双备份
- Refresh Token: 长效token,用于自动续期
- 续期策略: 距过期<60s时主动调用`POST /users/refresh`
- 并发安全: 使用`in-flight Promise`防止多个请求同时触发续期
- 401处理: 自动尝试续期 → 成功则重试原请求 → 失败则跳转登录页

### 2.4 前端职责总结

✅ **前端负责**:
- UI渲染和用户交互(Liquid Glass设计系统)
- 数据请求和展示(HTTP拦截/Token续期/数据格式化)
- 本地状态管理(Vuex Store/页面data/Storage持久化)
- 路由导航(uni-app路由/登录后回跳/TabBar切换)
- 权限校验(前端层面友好提示)
- SSE流式输出解析(AI聊天逐字显示)
- 图片懒加载和预览

❌ **前端不负责**:
- 业务逻辑验证(由后端完成)
- 权限最终判定(后端API层必须再次校验)
- 数据持久化(除本地缓存外)
- 复杂数据计算(由后端返回结果)

**关键原则**: 前端权限仅用于"用户体验优化",后端必须进行最终权限校验。

### 3.2 业务逻辑分析

#### **核心业务流程**

**1. 用户认证流程**:
```
登录: 接收LoginDTO → 查询用户 → Demo模式检查 → 账户锁定检查 → BCrypt校验 
    → 生成JWT → 返回LoginVO

注册: 验证短信验证码 → 检查手机号是否已注册 → BCrypt加密密码 → 初始化用户数据 
    → 插入users表 → 返回UserVO

实名认证: 获取当前用户ID → 更新certificationStatus为PENDING → (后台审核) 
       → 审核通过后更新为CERTIFIED
```

**2. 车源发布流程**:
```
发布: 获取当前用户ID → 验证CarCreateDTO → 根据pricingType区分(FIXED/AUCTION) 
    → 保存车源基本信息 → 保存图片 → 保存检测信息 → 保存标签 
    → isDraft=true设为DRAFT,否则设为ON_SALE → 返回CarVO
```

**3. 订单流转流程**:
```
创建订单 → 确认订单 → 支付保证金 → 合同签署 → 交易中 → 完成/终止

Step 1 创建: 验证车源存在且为FIXED → 生成订单号 → 创建Order记录(status=PENDING_CONFIRM)
Step 2 确认: 更新status=TRADING → 自动生成合同
Step 3 付保证金: 从买家保证金账户冻结depositAmount → 更新order(buyerDepositPaid=true)
Step 4 合同签署: submitContract → confirmContract → 双方都签署后status=SIGNED
Step 5 完成: 结算保证金 → 更新status=COMPLETED → 更新车源状态为SOLD
Step 6 取消/终止: 更新status=CANCELLED/TERMINATED → 退还保证金
```

**4. 拍卖流程**:
```
创建拍卖 → 竞拍中 → 出价 → 结束 → 结算 → 生成订单

Step 1 创建: 验证车源 → 验证endTime > startTime → 创建Auction记录
Step 2 出价: Redis分布式锁 → 验证状态/出价者/出价金额 → 标记之前最高出价为非中标 
          → 插入新出价记录 → 更新Auction(currentPrice,version++) → 释放锁
Step 3 结算: 验证状态为ENDED → 查找最高出价者 → 验证是否达到reservePrice 
          → 创建订单 → 更新Auction(status=SETTLED,winnerId,winningPrice) → 通知买家
```

**5. AI服务调用流程**:
```
普通对话: 接收Map params → 解析action字段 → 构建AI请求 → 返回AI响应

流式对话(SSE): 创建SseEmitter(超时5分钟) → 存入activeEmitters ConcurrentHashMap 
            → 异步调用AI服务 → 逐块接收响应并通过emitter.send推送 
            → event: message/done/error → 完成后emitter.complete()并从activeEmitters移除
```

### 3.4 技术架构要点

#### **1. 四层架构**

```
Controller层 → Service层 → Manager层 → Mapper层
```

- **Controller层**: 接收HTTP请求/参数校验/权限控制/响应封装
- **Service层**: 业务逻辑处理/事务管理/数据转换
- **Manager层**: 跨服务协调/第三方集成/消息队列处理(MessageProducer/MessageConsumer/PushManager)
- **Mapper层**: 数据库CRUD操作(MyBatis-Plus)

#### **2. 缓存策略(Redis使用场景)**

- **Token黑名单**: `token:blacklist:{tokenId}` → userId,TTL=Token剩余有效时间
- **拍卖出价锁**: `auction:bid:lock:{auctionId}` → userId,TTL=10秒,分布式锁防止并发出价
- **其他场景**: 用户会话缓存/热点车源数据缓存/验证码缓存

#### **3. 消息队列(RocketMQ使用场景)**

- **message-system**: 系统消息(订单状态变更/合同签署提醒/保证金变动通知)
- **message-chat**: 聊天消息(实时聊天消息推送,顺序消费)

**消息处理流程**:
```
Service层创建Message实体 → 调用MessageProducer.sendXXX(message) 
→ RocketMQ序列化Message为JSON → Consumer反序列化并转换为MessageVO 
→ 调用PushManager.pushToUser(userId,messageVO) → WebSocket推送给在线用户
```

#### **4. WebSocket实时通信**

- **端点**: `/ws`(支持SockJS降级)
- **协议**: STOMP over WebSocket
- **Broker**: SimpleBroker(内存消息代理)
- **前缀**: 应用目标`/app`,用户目标`/user`,广播主题`/topic`,`/queue`

**推送方式**:
```java
// 点对点推送(用户专属)
pushManager.pushToUser(userId, messageVO);

// 广播推送(全站公告)
pushManager.pushToAll(messageVO);
```

#### **5. JWT认证机制**

- **AccessToken**: 有效期24小时,载荷:userId/phone/nickname/roles/exp/iat
- **RefreshToken**: 有效期7天,用于换取新的AccessToken
- **认证流程**: 客户端请求携带Authorization: Bearer {token} → AuthenticationInterceptor拦截 
              → 解析token(验证签名/有效期/黑名单) → 将用户信息存入SecurityContext 
              → Controller通过SecurityUtils.getCurrentUserId()获取用户ID

#### **6. 事务管理**

- **声明式事务**: `@Transactional`注解
- **应用场景**: 订单创建/拍卖出价/保证金充值
- **传播行为**: REQUIRED(加入现有事务或创建新事务)

### 4.3 原型中的特殊交互逻辑

#### **交互逻辑1: 车行审核流程**
```
移动端提交入驻申请 
  → users.certification_status=PENDING
  → 运营端审核列表展示
  → 运营端通过 → UPDATE users SET certification_status='CERTIFIED'
  → INSERT INTO messages (通知用户)
  → PUBLISH Redis channel: user:certification:{userId}
  → 移动端接收 WebSocket 推送 / 下次请求刷新
```

**检查要点**:
- 审核通过后用户是否收到系统消息
- Redis Pub/Sub事件是否正确发布
- 移动端缓存是否正确清除

---

#### **交互逻辑2: 车源状态同步**
```
运营端下架车源 
  → UPDATE car_sources SET status='INACTIVE'
  → PUBLISH Redis channel: event:car:update:{carId}
  → 移动端后端监听 → 清除缓存
  → WebSocket 推送给卖家(可选)
```

**检查要点**:
- 下架后移动端列表是否立即不展示
- 收藏/浏览历史中的车源是否显示"已下架"
- 缓存清除是否及时

---

#### **交互逻辑3: 保证金变动通知**
```
运营端手动调整保证金 
  → UPDATE deposit_accounts SET balance=?
  → INSERT INTO deposit_records (流水记录)
  → PUBLISH Redis channel: event:deposit:change:{userId}
  → 移动端 WebSocket 推送
  → 用户余额实时更新
```

**检查要点**:
- 保证金变动后用户是否收到通知
- 余额显示是否与实际一致
- 流水记录是否完整

---

#### **交互逻辑4: 订单状态流转**
```
PENDING_CONFIRM(待确认)
  → 买家支付保证金 → TRADING(交易中)
  → 确认收货 → COMPLETED(已完成)
  
异常情况:
  → 发起争议 → DISPUTE(争议中)
  → 运营端处理 → COMPLETED/CANCELLED
  
  → 取消订单 → CANCELLED(已终止)
```

**检查要点**:
- 状态流转是否符合业务规则
- 保证金冻结/解冻时机是否正确
- 争议处理流程是否完整

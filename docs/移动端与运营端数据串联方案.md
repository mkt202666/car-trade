# 移动端与运营端数据串联方案

> **项目名称**: 5D好车 B2B二手车交易平台
> **版本**: v1.0.0
> **编写日期**: 2026-06-13
> **适用范围**: 移动端（uni-app）+ 运营端（React）双端数据集成

---

## 一、概述

### 1.1 项目背景

5D好车是一个B2B二手车交易平台，面向车商（车行/个人卖家）和买家提供二手车发布、浏览、交易等全链路服务。平台包含两个前端应用：

| 应用 | 技术栈 | 端口 | API 前缀 | 用户群体 |
|------|--------|------|----------|----------|
| **移动端** | uni-app（H5/小程序/App） | 8080 | `/api/v1` | 车商、买家 |
| **运营端** | React + Vite + Ant Design 5.x | 8081 | `/api/v1/admin` | 平台运营人员 |

两端共享同一个 PostgreSQL 数据库和 Redis 缓存实例，通过不同的后端服务（移动端后端 :8080、运营端后端 :8081）分别提供 API。

### 1.2 目标

1. **数据贯通**: 移动端产生的业务数据，运营端可实时查看和管理
2. **双向联动**: 运营端的操作（审核、下架、冻结等）能即时影响移动端用户体验
3. **统一标准**: 两端 API 遵循统一的响应格式、错误码、分页规范
4. **数据安全**: 运营端操作需审计追踪，移动端敏感数据需脱敏展示

### 1.3 涉及系统

| 系统 | 职责 | 技术栈 |
|------|------|--------|
| 移动端后端 | 移动端 API 服务 | Spring Boot 3.5.14 / Java 21 / MyBatis-Plus / RocketMQ / WebSocket |
| 运营端后端 | 运营管理 API 服务 | Spring Boot 3.5.14 / Java 21 / MyBatis-Plus / Redis |
| PostgreSQL | 共享数据存储 | PostgreSQL 16 |
| Redis | 共享缓存/消息 | Redis 7 |
| 移动端前端 | 用户端界面 | uni-app |
| 运营端前端 | 管理界面 | React + Vite + Tailwind CSS + Ant Design 5.x |

### 2.2 移动端后端 vs 运营端后端关系

| 维度 | 移动端后端 (`:8080`) | 运营端后端 (`:8081`) |
|------|----------------------|----------------------|
| 部署形态 | 独立 Spring Boot 应用 | 独立 Spring Boot 应用 |
| API 前缀 | `/api/v1` | `/api/v1/admin` |
| 认证方式 | JWT (Bearer Token) | JWT (Admin Bearer Token) |
| 用户体系 | `users` 表（车商/买家） | `admin_users` 表（运营人员） |
| 数据库 | 共享 PostgreSQL | 共享 PostgreSQL |
| Redis | 共享 Redis（DB 0） | 共享 Redis（DB 0） |
| 消息推送 | WebSocket + RocketMQ | 通过 Redis Pub/Sub 通知移动端 |
| 审计日志 | 操作日志（业务维度） | 审计日志（管理维度） |
| 端口 | 8080 | 8081 |

### 2.3 共享数据库策略

两端后端连接同一个 PostgreSQL 实例 (`new_car_trade`)，遵循以下策略：

| 策略 | 说明 |
|------|------|
| **读写分离** | 无（单实例，均为读写） |
| **表隔离** | 33张核心业务表共享；运营端独有 `admin_users`、`audit_logs`、`banners`、`export_templates`、`configs` 等表 |
| **事务隔离** | 两端不跨应用事务；同一应用内使用 Spring `@Transactional` |
| **缓存共享** | Redis DB 0 共享；运营端修改数据后主动清除相关缓存，移动端下次请求命中新数据 |
| **事件通知** | 运营端写操作 → Redis Pub/Sub → 移动端监听 → 清缓存/推 WebSocket |
| **锁策略** | 乐观锁（version 字段）+ 数据库行锁（`SELECT ... FOR UPDATE`） |

## 三、数据流转关系分析

### 3.1 按业务模块分析两端数据关系

#### 3.1.1 用户模块

| 移动端操作 | 涉及表 | 运营端可见数据 | 运营端可执行操作 |
|-----------|--------|---------------|-----------------|
| 注册/登录 | `users` | 用户列表、用户详情 | 查看用户、搜索、筛选、禁用/启用、编辑 |
| 实名认证 | `users.certification_status` | 认证状态列表 | 查看认证信息 |
| 设置车行名 | `users.shop_name` | 车行名称 | 查看、编辑 |
| 更新头像/昵称 | `users.avatar_url, users.nickname` | 最新头像和昵称 | 实时可见 |

#### 3.1.2 车行入驻与审核

| 移动端操作 | 涉及表 | 运营端可见数据 | 运营端可执行操作 |
|-----------|--------|---------------|-----------------|
| 提交入驻申请 | `users.certification_status = PENDING` | 待审核列表（badge: N） | 审核通过、审核拒绝（填写原因） |
| 车行信息填写 | `users.shop_name, users.real_name` | 申请详情 | 预览申请材料 |
| 审核通过通知 | `messages`（系统消息） | — | 系统自动推送通知 |

**审核流程数据流**:
```
移动端提交 → users.certification_status=PENDING
                    ↓
运营端审核列表 → SELECT * FROM users WHERE certification_status='PENDING'
                    ↓
运营端通过 → UPDATE users SET certification_status='CERTIFIED'
         → INSERT INTO messages (通知用户)
         → PUBLISH Redis channel: user:certification:{userId}
                    ↓
移动端接收 → WebSocket 推送 / 下次请求刷新
```

#### 3.1.3 车源管理

| 移动端操作 | 涉及表 | 运营端可见数据 | 运营端可执行操作 |
|-----------|--------|---------------|-----------------|
| 发布车源 | `car_sources, car_images, car_inspections, car_tags` | 车源列表（全平台） | 查看、编辑、下架、删除、标记违规 |
| 上架/下架 | `car_sources.status` | 车源状态 | 运营端可强制下架 |
| 修改车源信息 | `car_sources.*` | 最新数据 | 实时可见 |

**车源状态流转**:
```
ACTIVE（在售） ──→ INACTIVE（已下架）──→ ACTIVE（重新上架）
     │                                     ↑
     └──→ SOLD（已售出）                   │
                                          │
     ←── 运营端强制下架 ←─────────────────┘
```

#### 3.1.4 交易流程

| 移动端操作 | 涉及表 | 运营端可见数据 | 运营端可执行操作 |
|-----------|--------|---------------|-----------------|
| 创建订单 | `orders` | 订单列表（全平台） | 查看订单详情 |
| 支付保证金 | `deposit_records, deposit_accounts` | 保证金流水 | 查看流水 |
| 确认/取消订单 | `orders.status, order_logs` | 订单状态变更 | 查看日志 |
| 发起争议 | `disputes` | 争议列表（badge: N） | 处理争议、判定结果 |

**订单状态流转**:
```
PENDING_CONFIRM ──→ TRADING ──→ COMPLETED
       │                │
       ↓                ↓
   CANCELLED        DISPUTE ──→ COMPLETED / CANCELLED
```

#### 3.1.5 保证金管理

| 移动端操作 | 涉及表 | 运营端可见数据 | 运营端可执行操作 |
|-----------|--------|---------------|-----------------|
| 充值保证金 | `deposit_accounts, deposit_records` | 余额及流水 | 手动充值/扣减（管理员操作） |
| 交易冻结 | `deposit_records (FREEZE)` | 冻结记录 | 查看 |
| 交易解冻/退款 | `deposit_records (UNFREEZE/REFUND)` | 解冻/退款记录 | 查看 |
| 保证金不足 | `deposit_accounts.balance < threshold` | 预警列表 | 查看预警 |

#### 3.1.6 资源配置（Banner/协议）

| 运营端操作 | 涉及表 | 移动端可见数据 | 影响方式 |
|-----------|--------|---------------|---------|

#### 3.1.7 车型库管理

## 四、数据交互接口规范

### 4.1 RESTful API 设计规范

| 规范项 | 说明 |
|--------|------|
| 基础路径 | `/api/v1/admin` |
| 认证方式 | JWT Bearer Token（Header: `Authorization: Bearer <token>`） |
| 请求格式 | `Content-Type: application/json` |
| 响应格式 | 统一 JSON 结构 `ApiResponse<T>` |
| HTTP 方法 | GET(查询)、POST(创建)、PUT(更新)、DELETE(删除) |
| URL 命名 | 全小写 + 中划线，资源名复数形式 |
| 版本管理 | URL 路径版本 `/api/v1/admin` |

### 4.2 统一响应格式

```json
{
  "code": 200,
  "message": "success",
  "data": {},
  "timestamp": 1718284800000,
  "traceId": "trace-abc-123"
}
```

### 4.3 分页规范

请求参数:
| 参数 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| page | Integer | 1 | 当前页码（从1开始） |
| size | Integer | 20 | 每页条数（最大500） |
| sortBy | String | createdAt | 排序字段 |
| sortDir | String | DESC | 排序方向（ASC/DESC） |

分页响应体:
```json
{
  "code": 200,
  "data": {
    "list": [],
    "total": 150,
    "page": 1,
    "size": 20,
    "totalPages": 8
  }
}
```

### 4.4 错误码规范

| 错误码 | HTTP 状态码 | 说明 |
|--------|-------------|------|
| 200 | 200 | 成功 |
| 400 | 400 | 请求参数错误 |
| 401 | 401 | 未认证（Token 无效/过期） |
| 403 | 403 | 无权限（非管理员角色） |
| 404 | 404 | 资源不存在 |
| 409 | 409 | 数据冲突（重复操作） |
| 422 | 422 | 业务校验失败 |
| 500 | 500 | 服务器内部错误 |

## 五、数据同步机制

### 5.1 实时同步方案

#### 5.1.1 Redis Pub/Sub 事件通知

运营端执行写操作后，发布 Redis 事件通知移动端后端清除缓存或推送消息：

| 频道名称 | 触发场景 | 通知内容 |
|----------|---------|---------|
| `event:car:update:{carId}` | 运营端编辑/下架车源 | `{carId, action: "OFFLINE"/"UPDATE"/"DELETE"}` |
| `event:user:certification:{userId}` | 审核通过/拒绝 | `{userId, status: "CERTIFIED"/"REJECTED", reason}` |
| `event:deposit:change:{userId}` | 手动调整保证金 | `{userId, amount, type}` |
| `event:banner:update` | Banner 增删改 | `{type: "BANNER"/"POPUP"}` |
| `event:vehicle:update` | 品牌车型库变更 | `{action: "ADD"/"UPDATE"/"DELETE"}` |
| `event:config:update:{key}` | 配置文本变更 | `{key}` |

**运营端事件发布（Java 示例）**:
```java
@Component
@RequiredArgsConstructor
public class AdminEventPublisher {

    private final StringRedisTemplate redisTemplate;

    public void publishCarUpdate(Long carId, String action) {
        Map<String, String> event = Map.of("carId", String.valueOf(carId), "action", action);
        redisTemplate.convertAndSend("event:car:update:" + carId,
                JsonUtils.toJson(event));
    }

    public void publishCertificationResult(Long userId, String status, String reason) {
        Map<String, String> event = Map.of(
                "userId", String.valueOf(userId),
                "status", status,
                "reason", reason != null ? reason : "");
        redisTemplate.convertAndSend("event:user:certification:" + userId,
                JsonUtils.toJson(event));
    }

    public void publishBannerUpdate(String type) {
        Map<String, String> event = Map.of("type", type);
        redisTemplate.convertAndSend("event:banner:update", JsonUtils.toJson(event));
    }
}
```

**移动端后端事件监听（Java 示例）**:
```java
@Component
@RequiredArgsConstructor
public class AdminEventListener {

    private final CarCacheService carCacheService;
    private final VehicleCacheService vehicleCacheService;
    private final PushManager pushManager;

    @RedisMessageListener(channels = {"event:car:update:*"})
    public void onCarUpdate(Message message, byte[] pattern) {
        String channel = new String(pattern);
        String body = new String(message.getBody());
        Map<String, String> event = JsonUtils.toMap(body);
        Long carId = Long.valueOf(event.get("carId"));

        // 清除车源缓存
        carCacheService.clearDetailCache(carId);
        carCacheService.clearListCache();
    }

    @RedisMessageListener(channels = {"event:user:certification:*"})
    public void onCertificationUpdate(Message message, byte[] pattern) {
        String body = new String(message.getBody());
        Map<String, String> event = JsonUtils.toMap(body);
        Long userId = Long.valueOf(event.get("userId"));
        String status = event.get("status");

        // 推送 WebSocket 通知给用户
        MessageVO msg = new MessageVO();
        msg.setType("SYSTEM");
        if ("CERTIFIED".equals(status)) {
            msg.setTitle("车行认证通过");
            msg.setContent("恭喜！您的车行认证已通过审核，现在可以发布车源了。");
        } else {
            msg.setTitle("车行认证未通过");
            msg.setContent("您的车行认证审核未通过，原因：" + event.get("reason"));
        }
        pushManager.pushToUser(userId, msg);

        // 清除用户缓存
        userCacheService.clearUserInfo(userId);
    }
}
```

#### 5.1.2 WebSocket 实时推送

移动端已集成 WebSocket（STOMP over SockJS），运营端操作可通过两种途径通知移动端：

| 途径 | 适用场景 | 延迟 |
|------|---------|------|
| Redis Pub/Sub → 移动端后端 → WebSocket | 需要推送给特定用户的操作（审核结果、保证金变动） | 毫秒级 |
| 移动端轮询 | 低频更新（Banner列表、车型库） | 秒级（30s 间隔） |
| 主动刷新 | 用户操作时（打开页面、下拉刷新） | 即时 |

### 5.2 数据一致性保障策略

| 策略 | 实现方式 | 适用场景 |
|------|---------|---------|
| **缓存失效** | 运营端写操作 → Redis Pub/Sub → 移动端清缓存 | 车源编辑、用户状态变更 |
| **事务保证** | 单应用内 `@Transactional`，跨应用不使用分布式事务 | 订单创建、保证金操作 |
| **最终一致性** | 运营端写 DB → 发事件 → 移动端异步处理 | Banner更新、车型库变更 |
| **乐观锁** | `version` 字段 + `WHERE version = ?` | 并发编辑同一资源 |
| **数据库约束** | UNIQUE 约束、外键约束、CHECK 约束 | 数据完整性 |
| **幂等设计** | 接口支持幂等（订单号唯一、审核状态机） | 防重复操作 |

### 5.3 缓存策略

#### 移动端缓存

| 缓存对象 | Key 格式 | 过期时间 | 更新策略 |
|----------|----------|----------|----------|
| 车源详情 | `car:detail:{id}` | 30分钟 | 写时删除 + Pub/Sub 通知 |
| 车源列表 | `car:list:{hash}` | 5分钟 | 写时全删 |
| 用户信息 | `user:info:{id}` | 1小时 | 写时删除 + Pub/Sub 通知 |

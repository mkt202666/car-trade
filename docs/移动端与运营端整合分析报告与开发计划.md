# 5D好车 移动端与运营端整合分析报告与开发计划

> **版本**: v1.0.0  
> **日期**: 2026-06-14  
> **范围**: 移动端后端(:8080) + 运营端后端(:8081) + 移动端前端(uni-app) + 运营端前端(React)

## 第一部分：业务流程梳理与整合

### 一、系统全景

5D好车平台由四个子系统组成：移动端后端（Spring Boot :8080）、运营端后端（Spring Boot :8081）、移动端前端（uni-app H5）、运营端前端（React Vite）。两端后端共享同一个 PostgreSQL 16 数据库（33张核心业务表 + 5张运营端独有表）和 Redis 7 实例，通过 Redis Pub/Sub 实现跨服务事件通知。

### 二、已实现的核心业务链路

#### 2.1 用户注册 → 运营端用户管理

移动端用户通过手机号+密码注册（或 demo 模式自动注册），写入 `users` 表。运营端通过 `AppUser` 实体映射同一张表，`AdminUserService.listUsers()` 提供分页查询、关键字搜索、状态/认证过滤。运营端可禁用/启用用户，禁用后通过 `mobile:events` 频道推送 `USER_STATUS_CHANGED` 通知，移动端 `MobileEventListener` 接收后通过 WebSocket 告知用户。

**链路状态**: ✅ 基本完整。数据查询、状态变更、通知推送全链路已通。

#### 2.2 车行入驻 → 运营端审核

移动端用户提交实名认证后，`certification_status` 变为 `PENDING`，同时 `user_role` 升级为 `SHOP`。运营端 `AdminShopReviewService` 查询待审核列表，审核通过后更新认证状态为 `CERTIFIED`，审核拒绝写入 `REJECTED` 原因。两种操作均发布 `SHOP_AUDIT_RESULT` 事件到 `mobile:events` 频道，同时插入系统消息到 `messages` 表。

**链路状态**: ✅ 基本完整。审核通过/拒绝 → 事件通知 → WebSocket 推送全链路已通。

#### 2.3 车源发布 → 运营端车源管理

移动端车行用户发布车源写入 `car_sources` 表（及 `car_images`、`car_tags`、`car_inspections` 关联表）。运营端 `AdminCarService` 提供列表查询（多维度筛选）、详情查看、状态变更（上下架）、违规标记、编辑、软删除功能。所有写操作均通过 `AdminNotificationService` 发布 `CAR_STATUS_CHANGED` 事件，移动端 `MobileEventListener` 接收后清除车源缓存并推送 WebSocket 消息。

**链路状态**: ✅ 基本完整。CRUD + 通知 + 缓存清除已实现，但车源详情缺少图片和检测报告查询（代码中有 TODO 标记）。

#### 2.4 交易订单 → 运营端交易管理

移动端创建订单写入 `orders` 表，订单状态流转包括 `PENDING_CONFIRM → TRADING → COMPLETED / CANCELLED / DISPUTE`。运营端 `AdminOrderService` 提供订单列表、详情、操作日志查询。

**链路状态**: ⚠️ 部分缺失。运营端仅支持只读查询，缺少订单审核、退款处理等主动干预能力。

#### 2.5 保证金管理

移动端用户的保证金账户存储在 `deposit_accounts` 表，流水记录在 `deposit_records` 表。运营端 `AdminDepositService` 提供账户列表、流水查询、手动调整（充值/扣减/冻结/解冻）、汇总统计功能。手动调整操作发布 `DEPOSIT_CHANGED` 事件通知用户。

**链路状态**: ✅ 基本完整。

#### 2.6 争议处理

移动端用户发起争议写入 `disputes` 表。运营端 `AdminDisputeService` 提供争议列表、处理（通过/驳回/协商）功能，处理完成后发布 `DISPUTE_RESOLVED` 事件。

**链路状态**: ✅ 基本完整。

#### 2.7 资源配置（运营端 → 移动端）

运营端 `AdminResourceService` 管理 Banner 广告（CRUD + 排序 + 启停）和系统配置文本（用户协议/隐私条款/交易规范/合同模板）。移动端通过 API 请求实时获取这些数据。

**链路状态**: ⚠️ 运营端后端已实现，但移动端后端缺少对应的 Banner/Config 读取接口（开发计划中 Day 16 的 7 个只读接口尚未开发）。

#### 2.8 车型库管理（运营端 → 移动端）

运营端 `AdminCarLibraryService` 提供品牌/车系/车型三级 CRUD 接口。移动端车源发布时依赖此数据。

**链路状态**: ⚠️ 运营端后端已实现，但移动端后端缺少车型字典的公开读取接口。

#### 2.9 求购管理

移动端用户发布求购写入 `purchase_demands` 表。运营端 `AdminPurchaseRequestService` 提供列表查询、详情查看功能。

**链路状态**: ✅ 基本完整。但求购匹配功能（运营端手动/自动匹配车源）尚未实现。

### 三、事件通知机制现状

运营端通过 `MobileEventPublisher` 向 Redis 频道 `mobile:events` 发布 `MobileNotification` 对象（包含 7 种通知类型）。移动端 `MobileEventListener` 通过 `@PostConstruct` 注册 Redis 监听器，接收消息后路由到对应的 handler 方法：清除相关缓存 + 通过 `PushManager` 推送 WebSocket 消息。

已验证的事件发布链路：
- `AdminShopReviewService` → `SHOP_AUDIT_RESULT` ✅
- `AdminCarService` → `CAR_STATUS_CHANGED` ✅
- `AdminDisputeService` → `DISPUTE_RESOLVED` ✅
- `AdminDepositService` → `DEPOSIT_CHANGED` ✅
- `AdminUserService` → `USER_STATUS_CHANGED` ✅

尚未接入的事件类型：
- `SYSTEM_ANNOUNCEMENT` — 系统公告广播（无发送方）
- `PURCHASE_MATCHED` — 求购匹配（无发送方）

## 第二部分：问题清单与风险评估

### 一、P0 — 严重问题（必须立即修复）

#### P0-1: AppUser 实体缺少 user_role 字段

运营端 `AppUser` 实体映射 `users` 表，但未包含最近新增的 `user_role` 字段。这导致运营端无法通过 `user_role` 筛选用户类型，`AdminUserService` 仍使用 `shop_name` 是否为空来区分车行/个人用户，与移动端的 `user_role` 方案不一致。

**影响**: 运营端用户筛选逻辑不准确，无法查询 DEVELOPER 角色用户。  
**修复**: 在 `AppUser` 实体中添加 `userRole` 字段，并将 `AdminUserService` 的 SHOP_USER/PERSONAL_USER 过滤条件改为基于 `user_role` 字段。

#### P0-2: 移动端缺少 Banner/Config/车型字典读取接口

运营端已实现 Banner、配置文本、车型库的管理功能，但移动端后端完全没有对应的只读接口。这意味着移动端前端无法获取首页 Banner 轮播、用户协议文本、车源发布时的品牌/车系/车型级联数据。

**影响**: 移动端首页 Banner 展示、协议页面、车源发布车型选择等核心功能无法正常工作。  
**修复**: 在 `car-trade-backend` 中新增 7 个只读接口（Banner 列表、弹窗广告、配置文本、品牌列表、车系列表、车型列表）。

#### P0-3: CarCacheService 使用 KEYS 命令清除列表缓存

`CarCacheService.clearListCache()` 使用 Redis `KEYS` 命令匹配模式删除缓存键。`KEYS` 是 O(N) 阻塞操作，在生产环境数据量大时会导致 Redis 性能严重下降甚至阻塞其他请求。

**影响**: 生产环境 Redis 性能风险。  
**修复**: 改用 `SCAN` + `DEL` 组合，或使用缓存键集合（Set）管理列表缓存键。

### 二、P1 — 重要问题（本迭代内修复）

#### P1-1: 车源详情缺少图片和检测报告查询

`AdminCarService.getCarDetail()` 中有两处 TODO 标记：图片和检测报告查询未实现。运营端查看车源详情时无法看到车辆图片和车况检测报告，严重影响运营审核能力。

**修复**: 添加 `AppCarImageMapper` 和 `AppCarInspectionMapper`，在详情查询中联查关联数据。

#### P1-2: AdminUserService 用户筛选应使用 user_role 字段

当前 `listUsers()` 使用 `shop_name IS NOT NULL` 判断车行用户，这与新增的 `user_role` 字段语义冲突。用户可能设置了 `user_role = SHOP` 但尚未填写 `shop_name`，或者反之。

**修复**: 将过滤条件改为 `user_role = 'SHOP'` / `user_role = 'PERSONAL'`。

#### P1-3: 运营端缺少订单主动干预能力

运营端 `AdminOrderService` 仅支持只读查询（列表、详情、日志），缺少以下运营必需的操作能力：
- 订单审核（待确认订单的强制审核通过）
- 退款处理（争议订单的保证金退款）
- 强制终止（异常订单的强制关闭）

**修复**: 添加订单审核、退款、强制终止等写操作接口。

#### P1-4: 移动端 AppUser 缺少多个用户表字段

`AppUser` 实体缺少以下 `users` 表中的字段：`viewCountToday`、`messageCount`、`messageCountToday`、`followerCountToday`、`memberExpireAt`、`description`、`userRole`、`notificationSettings`、`loginFailCount`、`lockedUntil`。运营端用户详情和统计中无法展示这些数据。

**修复**: 补齐 `AppUser` 实体缺失字段。

#### P1-5: MobileEventListener 系统公告广播未实现

`handleSystemAnnouncement()` 方法中，当 `targetUserId` 为 null 时（即真正的广播场景），仅记录日志而未推送给任何用户。代码中有 TODO 标记但未实现。

**修复**: 实现广播机制，可通过维护在线用户列表或使用 STOMP 的 `/topic/announcements` 广播目的地。

#### P1-6: 移动端与运营端分页/响应格式不一致

- 移动端 `ApiResponse` 使用 `code: 200` 表示成功
- 运营端 `ApiResponse` 使用 `code: 0` 表示成功
- 移动端 `PageResult` 包含 `totalPages` 字段，运营端不包含

两端前端如果后续需要共享工具函数或进行统一处理，这种不一致会造成困扰。

**修复**: 统一为 `code: 200` 表示成功（或统一为 `code: 0`），`PageResult` 结构对齐。

### 三、P2 — 改进项（下一迭代处理）

#### P2-1: 缺少乐观锁实现

设计文档中提到了乐观锁（version 字段），但 `AppCarSource`、`AppUser` 等实体均未包含 `@Version` 字段。`auctions` 表有 `version` 列但其他表没有。两端并发操作同一数据时可能出现覆盖写入。

#### P2-2: 缺少车源违规记录表

`AdminCarService.violateCar()` 中违规信息仅通过 `log.info()` 记录，未持久化到数据库。缺少独立的违规记录表，无法追踪历史违规信息和进行统计分析。

#### P2-3: 运营端前端 API 层缺失

`car-trade-admin-frontend` 的页面骨架已完成（16 个页面目录），但缺少 `src/api/` 层的 API 调用封装。前端页面目前使用静态 mock 数据，需要对接后端 API。

#### P2-4: 缺少数据导出功能实现

`ExcelExportUtil` 工具类已存在，`ExportTemplate` 实体和 Mapper 已创建，但缺少实际的导出 Controller/Service 实现。

#### P2-5: Dashboard 聚合查询未优化

`DashboardService` 需要进行多表联查（users + car_sources + orders + deposit_accounts），当数据量增长后可能出现慢查询。缺少缓存策略和预计算机制。

#### P2-6: 缺少审计日志查询接口

`AuditLogController` 和 `AuditLogService` 已存在，但功能完整性待验证。运营端操作应全部记录审计日志并支持查询。

### 四、P3 — 潜在风险

#### P3-1: 城市数据质量问题

`data-cities.sql` 中存在重复条目（武汉/广州/深圳/成都/重庆出现两次）、城市编码不一致（"ZHUHai" vs "Zhuhai"、"XI'AN" vs "XIAN"）。前端城市选择和后端城市查询可能出现不匹配。

#### P3-2: JWT Secret 安全

运营端 JWT Secret 已移除默认值，必须通过环境变量注入。但两端共享同一密钥的机制需要在部署文档中明确说明，避免配置错误导致安全漏洞。

#### P3-3: RocketMQ 可选性

`MessageConsumer` 和 `MessageProducer` 使用 `@ConditionalOnBean` 条件加载，如果 RocketMQ 未配置则不加载。但部分业务逻辑（如消息推送）可能依赖 RocketMQ 消费者，需要确保降级路径完整。

#### P3-4: 设计文档与代码不同步

`04-数据库设计.md` 引用 28 张表，实际 `init.sql` 有 33 张表。多处 ALTER TABLE 新增的字段未反映在设计文档中。建议以 `init.sql` 为唯一真相来源。

### 四、测试验证方案

#### 4.1 单元测试策略

当前两端后端均无测试代码。建议至少为以下核心 Service 添加单元测试：

- `AdminUserService.listUsers()` — 验证 user_role 过滤逻辑
- `AdminCarService.updateCar()` / `deleteCar()` — 验证类型转换和通知发送
- `AdminShopReviewService.approve()` / `reject()` — 验证状态变更和事件发布
- `MobileEventListener.handleNotification()` — 验证消息路由和缓存清除

#### 4.2 接口集成测试

使用 Postman/Apifox 集合覆盖全部 70+ 运营端接口 + 7 个移动端新增接口。重点验证：
- 认证流程（登录 → Token → 刷新 → 过期处理）
- 分页参数边界值（page=0, size=0, size>500）
- 权限控制（无 Token / 过期 Token / 非管理员 Token）
- 数据一致性（运营端修改后移动端查询结果同步）

#### 4.3 端到端链路测试

| 测试场景 | 验证点 |
|---------|--------|
| 车行审核通过 | 运营端审核 → 用户认证状态变更 → WebSocket 消息到达 → 移动端刷新可见 |
| 车源违规下架 | 运营端标记违规 → 车源状态变更 → 缓存清除 → 车主收到通知 |
| 保证金手动调整 | 运营端充值/扣减 → 余额变更 → 流水记录 → 用户收到通知 |
| Banner 配置 | 运营端添加 Banner → 移动端接口返回新 Banner → 首页展示 |
| 用户禁用 | 运营端禁用用户 → 用户状态变更 → 下次登录被拒 |

**分层职责边界**:

| 层 | 职责 | 禁止行为 |
|----|------|----------|
| **Controller** | 参数校验、路由分发、调用 Service | 不得包含业务逻辑、不得直接操作数据库 |
| **Service** | 业务逻辑编排、事务控制 | 不得持有 HttpServletRequest/Response |
| **Manager** | 组合多个 Mapper 调用、调用第三方服务、缓存编排（被 Service 调用） | 不得包含跨 Service 的业务逻辑、不得被 Controller 直接调用 |
| **Mapper** | 数据库访问，继承 BaseMapper | 不得包含业务逻辑 |
| **Entity** | 数据库表映射，仅含字段定义 | 不得包含业务方法 |
| **Converter** | Entity ↔ DTO ↔ VO 对象转换 | 不得包含业务逻辑、不得操作数据库 |
| **DTO** | 请求参数封装，可含校验注解 | 不得复用为响应对象 |
| **VO** | 响应数据封装，按需裁剪字段 | 不得包含数据库注解 |

**类名**:

| 元素 | 规约 | 示例 |
|------|------|------|

| VO | 名词/名词 + VO | `CarVO`, `CarDetailVO` |

| 拦截器 | 名词 + Interceptor | `AuthInterceptor` |

**方法名**:

| 方法 | 命名规则 | 示例 |
|------|----------|------|
| 新增 | create / save | `create(CarCreateDTO)` |
| 删除 | delete / remove | `delete(Long id)` |

| 查询单个 | get / find | `getById(Long id)` |

**Converter 方法**:

| 转换方向 | 命名规则 | 示例 |
|----------|----------|------|

**常量**:
- 全大写 + 下划线分词，如 `MAX_UPLOAD_SIZE`
- 枚举值全大写 + 下划线，如 `PENDING_CONFIRM`

**包名**:
- 全部小写，单数形式，如 `com.pancosky.newcartrade.service`

### 1.3 MyBatis-Plus 使用规约

1. **所有 Mapper 继承 BaseMapper**，不手写重复 CRUD
2. **逻辑删除**：统一使用 `deletedAt` 字段（TIMESTAMP 类型，非 null = 已删除）
3. **分页**：统一使用 MyBatis-Plus `Page` 对象 + `PaginationInnerInterceptor`
4. **复杂查询**：
   - 简单查询：用 `LambdaQueryWrapper`
   - 多表关联 / 复杂报表：写 XML Mapper + 自定义 ResultMap
   - 禁止在 Service 层拼接 SQL
5. **条件构造器规范**：
   - 使用 `LambdaQueryWrapper` 避免字段名硬编码
   - 多条件组装抽取为 `buildQueryWrapper` 方法
   - `eq` 前判 null，避免空条件查询全表
6. **自动填充**：`createdAt` / `updatedAt` 通过 `MetaObjectHandler` 统一处理，禁止手动 set
7. **枚举处理**：所有数据字典实现 `IEnum<Integer>`，避免存字符串

### 1.4 异常处理规约

1. **业务异常**：抛出 `BusinessException(String message)`，统一在全局异常处理器捕获
2. **Controller 层不做 try-catch**，异常向上抛至全局处理器
3. **全局异常处理器**（`@RestControllerAdvice`）：
   - `BusinessException` → 400 + 错误消息
   - `MethodArgumentNotValidException` → 400 + 校验失败详情
   - `Exception` → 500 + 不泄露堆栈（日志中记录完整堆栈）
4. **统一响应**：全部返回 `ApiResponse<T>`，禁止直接返回 Entity

### 1.5 日志规约

1. **使用 Lombok `@Slf4j`**，禁止手动声明 Logger
2. **日志级别**：
   - ERROR：系统异常、数据库错误、调用第三方失败
   - WARN：参数异常、可恢复的错误
   - INFO：关键业务操作（创建订单、支付、发布车源）
   - DEBUG：开发调试用，禁止在生产打印
3. **日志内容**：包含操作人、操作类型、关键业务 ID，格式 `"操作: 用户{}{}车源{}"`
4. **禁止打印**：密码、Token、身份证号、手机号等敏感信息

### 1.6 数据库设计规约补充

1. **表名**：全小写 + 下划线，复数形式（`car_sources`, `order_logs`）
2. **字段名**：全小写 + 下划线（`user_id`, `created_at`）
3. **主键**：BIGSERIAL AUTO_INCREMENT，统一 `id` 字段
4. **时间字段**：统一 `TIMESTAMP` 类型，字段名 `_at` 后缀（`created_at`, `updated_at`）
5. **逻辑删除**：`deleted_at TIMESTAMP`，NULL = 未删除
6. **所有表必须包含**：`id`, `created_at`, `updated_at`
7. **索引**：外键字段必须建索引，高频查询字段建索引，避免冗余索引

### 1.7 消息推送规约

1. **实时推送**：采用 STOMP over WebSocket，通过 `/user/queue/messages` 点对点推送给指定用户
2. **异步解耦**：所有消息先发 RocketMQ，再由 Consumer 消费后落库 + 推送，不直推 WebSocket
3. **Topic 划分**：
   - `message-system` — 系统通知（审核结果、交易提醒），顺序不敏感
   - `message-chat` — 聊天消息，按 receiverId hash 有序消费保证同一会话顺序
4. **失败处理**：MQ Consumer 消费失败重试 3 次，仍失败写入 error log + 报警
5. **离线消息**：WebSocket 断开不丢消息；消息已落库，用户下次登录后主动拉取未读消息
6. **连接鉴权**：WebSocket 握手时通过 `Authorization` header 校验 JWT，未认证断开连接
7. **不跨进程推送**：分布式部署时每个实例只推送自己 WebSocket 连接上的用户，跨实例通过 RocketMQ 广播转发（预留）

### 2.3 组件层级边界

```
页面组件（pages/）
  └── 容器组件（负责数据获取、状态管理、布局）
       └── 展示组件（components/，纯 UI，通过 props 接收数据，通过 events 通信）
```

**页面组件职责**：
- 调用 API 获取数据
- 管理页面级别状态
- 布局和路由跳转

**展示组件职责**：
- 只通过 `props` 接收数据
- 通过 `$emit` 通知父组件
- 不直接调用 API
- 不访问 store

### 2.4 请求封装规范

1. **统一请求实例**：在 `api/request.js` 中创建，配置 baseURL、超时时间、拦截器
2. **请求拦截器**：自动注入 JWT Token、添加平台标识（h5/mp/app）
3. **响应拦截器**：统一处理 401 跳登录、500 提示错误、业务错误码处理
4. **API 模块**：按业务模块拆分到 `api/*.js`，每个函数返回 Promise
5. **命名**：`getList`, `getDetail`, `create`, `update`, `remove`

```javascript
// api/car.js 示例
import request from './request'

export const getCarList = (params) => request.get('/cars', { params })
export const getCarDetail = (id) => request.get(`/cars/${id}`)
export const createCar = (data) => request.post('/cars', data)
export const favoriteCar = (id) => request.post(`/cars/${id}/favorite`)
```

### 2.5 状态管理规范

1. **优先使用 props + events 通信**，页面组件之间用 uni-app 的 `EventBus` 或 uni.$on
2. **全局状态（用户信息、Token）**放 `store/`
3. **服务端数据**不存入 store，直接在页面组件中请求和管理
4. **避免滥用 Vuex/Pinia**——大多数页面级数据不需要全局共享

### 2.6 uni-app 多平台兼容规约

1. **条件编译**：多平台差异用条件编译处理，而不是运行时判断

```vue
<!-- #ifdef MP-WEIXIN -->
<button open-type="getPhoneNumber">获取手机号</button>
<!-- #endif -->
<!-- #ifdef APP-PLUS -->
<button @click="getPhoneByApp">获取手机号</button>
<!-- #endif -->
```

2. **尽量使用 uni 前缀 API**（uni.request / uni.navigateTo），避免平台特有 API
3. **样式**：使用 flex 布局，避免平台差异大的 CSS 属性
4. **字体图标**：使用 iconfont，兼容各平台
5. **页面单位**：使用 rpx 或 upx（uni-app 响应式单位）

## 三、通用规约

### 3.1 Git 分支规范

```
main             生产分支，只从 release 合并
release/         预发布分支，从 develop 合并
  └── v1.0.0
develop          开发主分支，从 feature 合并
  └── feature/    功能分支
        ├── feat-auth              新功能
        ├── fix-price-display      修复
        └── refactor-car-list      重构
```

| 分支 | 命名 | 合并目标 | 说明 |
|------|------|----------|------|
| 功能开发 | `feat-<功能名>` | develop | 新功能、新页面 |
| 修复 | `fix-<问题描述>` | develop | Bug 修复 |
| 重构 | `refactor-<模块>` | develop | 不改变行为的重构 |
| 发布 | `release/v<版本号>` | main + develop | 版本发布 |

**提交信息格式**：

```
<type>(<scope>): <简短描述>

<可选详细描述>
```

| Type | 场景 |
|------|------|
| feat | 新功能 |
| fix | 修复 |
| refactor | 重构 |
| docs | 文档 |
| style | 格式调整（无代码变化） |
| chore | 构建/工具变化 |

### 3.2 版本号管理

遵循语义化版本 `主版本.次版本.修订号`：

| 版本位 | 变动 | 示例 |
|--------|------|------|
| 主版本 | 不兼容的 API 修改 | 1.0.0 → 2.0.0 |
| 次版本 | 向下兼容的新功能 | 1.0.0 → 1.1.0 |
| 修订号 | 向下兼容的 bug 修复 | 1.0.0 → 1.0.1 |

## 四、强制检查清单

### 后端 PR 检查

- [ ] 无 `System.out.println`，使用 `@Slf4j`
- [ ] 敏感信息（密码/Token/手机号）未被日志打印
- [ ] 请求参数校验使用 `@Validated`
- [ ] 所有 DTO/VO 字段使用包装类型（Long/Integer），非基本类型
- [ ] 数据库字段判空使用 `wrapper.eq(条件, 字段, 值)`。其中条件不为 null 时才拼接
- [ ] 事务方法标记 `@Transactional(rollbackFor = Exception.class)`
- [ ] Controller 返回 `ApiResponse<T>`，不直接返回 Entity
- [ ] 消息推送通过 `PushManager` 封装，不直接调用 `SimpMessagingTemplate`
- [ ] 系统消息 / 聊天消息应走 RocketMQ 异步推送，不直推 WebSocket
- [ ] 浏览记录/操作日志使用 `@Async` 异步处理，不阻塞主流程
- [ ] 敏感数据掩码：手机号中间4位掩码、身份证中间10位掩码

### 前端 PR 检查

- [ ] 组件通过 `props` 接收数据，不直接调用 API
- [ ] API 请求通过 `api/*.js` 统一封装，不在页面中直接 `uni.request`
- [ ] 页面路由使用 `uni.navigateTo`，不直接操作 `window.location`
- [ ] 图片资源使用相对路径或 CDN，不硬编码地址
- [ ] 错误状态有 UI 反馈（Toast / 空状态 / 重试按钮）
- [ ] 列表页有加载状态和空状态处理

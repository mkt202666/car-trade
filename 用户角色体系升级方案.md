## 一、背景与问题

### 原有设计缺陷

之前通过 `shop_name` 字段判断用户类型存在以下问题：

1. **语义不清晰** - `shop_name` 是业务字段，不应承担角色判断职责
2. **扩展性差** - 无法支持更多用户类型（如开发人员、测试账号等）
3. **数据冗余** - 个人用户的 `shop_name` 始终为 NULL，浪费存储空间
4. **查询低效** - 需要 `IS NOT NULL AND != ''` 双重判断，无法有效利用索引
5. **维护困难** - 角色逻辑散落在多处代码中，容易遗漏

### 新设计方案

引入独立的 **`user_role`** 字段，明确标识用户角色：

```sql
ALTER TABLE users 
ADD COLUMN user_role VARCHAR(20) DEFAULT 'PERSONAL';
```

## 二、角色定义

### UserRole 枚举

```java
public enum UserRole {
    PERSONAL("PERSONAL", "个人用户"),      // 普通C端用户
    SHOP("SHOP", "车行用户"),              // B端认证商家
    ADMIN("ADMIN", "系统管理员"),          // 运营端后台管理员
    DEVELOPER("DEVELOPER", "开发人员");    // 内部测试账号
}
```

### 角色对比表

| 角色 | 说明 | shop_name | certification_status | 数据来源 |
|------|------|-----------|---------------------|---------|
| **PERSONAL** | 个人用户 | NULL/空 | NONE/PENDING/CERTIFIED | users 表 |
| **SHOP** | 车行用户 | 非空字符串 | PENDING/CERTIFIED/REJECTED | users 表 |
| **ADMIN** | 系统管理员 | NULL | NULL | admin_users 表 |
| **DEVELOPER** | 开发人员 | NULL/自定义 | NULL | users 表（特殊标记） |

## 八、注意事项

### 8.1 向后兼容

- ✅ **保留 shop_name 字段** - 现有业务逻辑继续使用
- ✅ **默认值为 PERSONAL** - 老数据自动归类为个人用户
- ✅ **自动同步逻辑** - 提交认证时自动升级为 SHOP

### 8.2 数据一致性

**关键规则**：
- `user_role = 'SHOP'` ⇔ `shop_name IS NOT NULL AND shop_name != ''`
- `user_role = 'PERSONAL'` ⇔ `shop_name IS NULL OR shop_name = ''`

**维护建议**：
- 在 Service 层统一维护这两个字段的同步
- 避免直接 SQL 更新导致不一致
- 定期运行数据一致性检查脚本
